# 推荐链路稳定性执行计划书

状态：执行基线  
日期：2026-04-17  
结论：本轮不再有“待确定项”。大批量导入可续跑 checkpoint 确定要做。

## 0. 一句话结论

这次要解决的不是单个按钮、单个缓存、单个页面的问题，而是整条推荐链路的稳定性：

```text
记录池 -> S 分数 -> 推荐选择 -> 当前三卡 -> 主页面 / 侧边栏 / widgets / 快捷复习 / 搜索动作
```

最终要做到：

- 多个页面、多个侧边栏、快捷键、搜索动作同时操作时，不互相覆盖状态。
- 当前三卡选择、游标推进、补卡、版本更新是一个完整后台动作。
- 搜索卡片里的按钮和推荐卡片按钮使用同一套后台规则。
- 大批量导入按批处理，每批成功后落永久池并写 checkpoint，崩溃后能继续。
- 不再单独做一个模糊的 “S 缓存健康检查” 项目。

## 1. 本轮目标

### 1.1 用户层目标

用户最终看到的是稳定的三张推荐卡。

这三张卡应该来自当前记录和 S 分数，而不是因为多页面同时打开、快捷键操作、搜索操作、批量导入中断导致：

- 三卡突然空掉
- 三卡重复
- 刚跳过的又回来
- 刚屏蔽的又展示
- 快捷复习和页面点击重复记录
- 搜索操作打乱当前推荐轮次
- 批量导入后长期处于半更新状态

### 1.2 工程层目标

本轮工程目标是四个工作包：

1. 关键推荐状态后台最终写。
2. 当前三卡选择原子化并加版本号。
3. 搜索按钮接入推荐卡片同一套动作系统。
4. 大批量导入 / 大批量书签变更做每批 checkpoint 可续跑任务。

同时明确删除一件事：

1. 不做独立的 S 缓存健康检查项目。

## 2. 本轮不做什么

这些内容不进入本轮主线：

- 不重新设计 S 公式。
- 不把 30 秒时间追踪 checkpoint 损失当作核心风险。
- 不新做广播系统。
- 不把搜索页面做成全局 S 池修复器。
- 不新增一个泛泛的 “S 缓存健康检查” 按钮或任务。
- 不做前端视觉重构。

## 3. 当前基础判断

代码里已经有一些基础能力：

- 大量书签事件会进入 bulk 模式，当前阈值约 120 个事件。
- 普通书签变更队列已经持久化，后台重启后可以恢复。
- 普通变更处理已经按 chunk 分段，当前 chunk 约 200。
- S 增量计算和精算里已有分批处理能力。
- 快捷复习已经在后台有入口。

但现状还缺三类关键保证：

1. 多入口同时写推荐状态时，没有完整收敛成“后台最终写”。
2. 当前三卡、游标、补卡、版本没有作为一个完整动作处理。
3. 导入后的恢复还不是完整的“每批 checkpoint 可续跑任务”。

## 4. 设计原则

### 4.1 前台负责交互，后台负责最终写入

前台仍然负责：

- 展示卡片
- 展示搜索结果
- 按钮 loading
- 错误提示
- 重试入口
- 本地临时 UI 状态

后台负责最终写入会影响推荐结果的状态。

原因：

Chrome storage 是共享存储，但它不理解推荐业务事务。它不会自动保证“跳过、稍后、屏蔽、翻牌、复习、当前三卡、游标”这些状态一起正确变化。

所以推荐业务状态必须由后台统一判断、统一写入、统一返回结果。

### 4.2 三卡状态必须有版本

每次后台成功改变当前三卡，就产生一个新版本。

前台发起动作时携带自己看到的版本。后台发现版本过期时，不让旧动作覆盖新状态，而是返回“状态已变化，需要刷新”。

### 4.3 搜索不是当前三卡轮次

搜索结果可以操作书签，但搜索结果本身不是当前三卡轮次。

所以：

- 展示搜索结果不会自动替换当前三卡。
- 搜索里点击提前复习、稍后、屏蔽，会更新这个书签和后续推荐。
- 如果搜索结果正好也是当前三卡之一，后台同步更新当前三卡。
- 如果搜索结果不在当前三卡里，不主动打乱当前轮次。

### 4.4 大批量任务允许当前批丢失，但不允许已完成批丢失

批量导入 / 大批量变更采用这个模型：

```text
当前批在内存处理
当前批成功
写入永久池
写入 checkpoint
进入下一批
```

如果浏览器崩溃：

- 当前批内存结果可以丢。
- 已完成批已经进入永久池。
- checkpoint 还在。
- 下次打开从 checkpoint 指向的位置继续。

关键要求：

- 已完成批不能丢。
- 当前批可以重跑。
- 重跑必须是幂等的，不能重复污染状态。

## 5. 工作包 A：关键推荐状态后台最终写

### 5.1 要解决的问题

多个入口可能同时改同一批推荐状态：

- 主推荐页
- 侧边栏
- widgets
- 快捷复习快捷键
- 搜索结果卡
- 多窗口里的插件页面

如果这些入口都能直接写推荐状态，就可能出现：

- 后写覆盖先写
- 旧页面覆盖新页面
- 翻牌和跳过交错
- 快捷复习和点击复习重复
- 当前三卡和游标不同步

### 5.2 需要后台最终写的状态

必须收敛到后台最终写：

- 当前三卡
- 推荐游标
- 跳过
- 稍后
- 屏蔽
- 翻牌
- 复习
- 快捷复习结果
- 会反向影响推荐结果的热力图状态

说明：

- 如果热力图只是展示统计，不影响 S 分、推荐池或三卡选择，它可以继续作为展示数据。
- 如果热力图会改变推荐结果，就必须走后台最终写。

### 5.3 前台允许保留的状态

这些可以留在前台：

- 当前搜索关键词
- 当前页面 tab
- 展开 / 折叠
- loading 状态
- toast 提示
- 临时筛选项
- 纯展示缓存

### 5.4 具体做法

把推荐相关按钮改成向后台发送动作。

动作包括：

- 打开
- 翻牌
- 标记已复习
- 待复习
- 稍后
- 屏蔽
- 跳过当前卡
- 刷新当前三卡
- 快捷复习下一张

后台收到动作后：

1. 读取当前推荐状态。
2. 判断动作是否仍然有效。
3. 写入对应状态。
4. 必要时重新计算这个书签的 S。
5. 必要时补卡或刷新当前三卡。
6. 生成新版本。
7. 返回最新结果给前台。

### 5.5 验收标准

- 前台不再直接写跳过、稍后、屏蔽、翻牌、复习、当前三卡、游标。
- 后台失败时，前台只显示失败或重试，不自己补写 storage。
- 主页面和侧边栏同时点同一个书签，最终状态一致。
- 快捷复习和页面按钮同时触发，不出现重复复习或重复翻牌。

## 6. 工作包 B：三卡选择原子化 + 版本号

### 6.1 要解决的问题

三卡选择不能拆成多个可交错步骤。

错误模型是：

```text
先选卡
再推进游标
再写当前三卡
中间任何一步都可能被另一个页面插进来
```

这样会导致：

- 当前三卡和游标脱节
- 两个页面互相覆盖
- 旧请求晚回来覆盖新状态
- 补卡补到重复项
- 当前轮次被跳过

### 6.2 正确模型

三卡选择必须是一个后台动作。

一个动作内完成：

1. 读取当前推荐池和状态。
2. 排除屏蔽、稍后、跳过、已处理项。
3. 选择三张卡或补足缺口。
4. 推进游标。
5. 写入当前三卡。
6. 写入新版本。
7. 返回最新三卡。

### 6.3 版本规则

当前三卡状态带版本。

前台发动作时携带版本：

- 版本匹配：后台执行动作。
- 版本过期：后台拒绝旧动作，返回最新状态或要求刷新。
- 无版本的旧入口：按兼容策略处理，但不能覆盖新状态。

### 6.4 补卡规则

不同动作对三卡影响不同：

- 跳过当前卡：移除该卡，补一张。
- 稍后当前卡：移除该卡，补一张。
- 屏蔽当前卡：移除该卡，补一张。
- 标记已复习当前卡：标记状态，必要时补卡。
- 搜索里操作非当前卡：只更新该书签，不补卡。
- 搜索里操作当前卡：按当前卡规则处理。
- 快捷复习：按当前三卡状态选择下一张，执行后更新当前三卡。

### 6.5 验收标准

- 两个页面同时刷新三卡，不会旧结果覆盖新结果。
- 当前三卡和游标始终匹配。
- 跳过、稍后、屏蔽后不会出现同一张卡立刻回来。
- 快捷复习不会和页面点击产生重复状态。

## 7. 工作包 C：搜索按钮接入推荐卡片同一套动作系统

### 7.1 要解决的问题

搜索结果不是当前三卡，但搜索结果里的按钮会改变推荐状态。

如果搜索按钮自己写状态，就会绕过推荐主链路。

如果搜索展示结果自动进入推荐轮次，就会打乱当前三卡。

### 7.2 搜索卡允许的按钮

搜索卡可以提供：

- 打开
- 提前复习 / 标记已复习
- 待复习
- 稍后
- 屏蔽

跳过按钮特殊处理：

- 如果搜索结果正好是当前三卡之一，可以显示跳过。
- 如果搜索结果不是当前三卡，不显示“跳过”，改用“稍后”或“近期不推荐”语义。

### 7.3 搜索动作规则

搜索按钮和推荐卡片按钮复用同一套后台动作。

区别只在于动作对象是否属于当前三卡：

```text
属于当前三卡：
  更新书签状态
  同步更新当前三卡
  必要时补卡

不属于当前三卡：
  更新书签状态
  必要时重新计算这个书签的 S
  不主动替换当前三卡
```

### 7.4 打开和复习的区别

打开不一定等于复习。

需要区分：

- 打开：只打开书签，后续由历史记录 / 时间记录自然影响 S。
- 打开并复习：打开并写入复习记录，立刻触发这个书签的 S 更新。
- 标记已复习：不一定打开页面，但写入复习记录并更新 S。

这样可以避免用户只是查看一下，却被系统当成复习。

### 7.5 搜索诊断规则

搜索可以展示诊断数据：

- 当前 S 分
- 分数来源
- 是否被屏蔽
- 是否稍后
- 是否待复习
- 是否可能进入推荐
- 记录是否不足

但搜索不负责修全局推荐池。

### 7.6 验收标准

- 搜索展示结果不会自动替换当前三卡。
- 搜索里提前复习一个非当前卡，只影响这个书签和后续推荐。
- 搜索里屏蔽当前卡，会同步从当前三卡移除并补卡。
- 搜索里屏蔽非当前卡，不打乱当前三卡。
- 搜索按钮和推荐卡片按钮的最终结果一致。

## 8. 工作包 D：大批量导入 / 大批量变更 checkpoint 可续跑

### 8.1 要解决的问题

当前已有 bulk、持久队列、分批计算基础，但导入后的完整恢复还不是“每批 checkpoint 可续跑任务”。

需要解决的是：

- 导入大量书签时，不要长期半更新。
- 浏览器崩溃后，不要从头覆盖。
- 当前批可以丢，但已完成批不能丢。
- 下次打开浏览器或插件时，能继续未完成任务。

### 8.2 任务状态

批量任务需要记录：

- 任务 id
- 任务类型：导入、批量新增、批量移动、批量删除
- 当前阶段
- 当前批次位置
- 已完成数量
- 总数量
- 最近成功时间
- 最近失败原因
- 是否需要继续

阶段可以分为：

1. 记录池补齐。
2. S 基础数据补齐。
3. S 分批计算。
4. 推荐池可用性恢复。
5. 当前三卡刷新或延迟刷新。

### 8.3 每批处理流程

每一批都按这个顺序：

```text
读取 checkpoint
确定下一批
在内存处理当前批
当前批处理成功
写入永久池
永久池写入成功
写入 checkpoint
进入下一批
```

重要顺序：

- 先写永久池。
- 再写 checkpoint。

原因：

如果先写 checkpoint 再写永久池，崩溃后系统会以为这批完成了，但数据其实没进去。

如果先写永久池再写 checkpoint，崩溃后最多重跑这一批。只要写入是幂等的，就不会污染结果。

### 8.4 当前批重跑规则

当前批可能在这些位置崩溃：

- 内存处理途中
- 永久池写入前
- 永久池写入后、checkpoint 写入前
- checkpoint 写入后、下一批开始前

要求：

- 永久池写入前崩溃：下次重跑这一批。
- 永久池写入后、checkpoint 前崩溃：下次可能重跑这一批，结果必须幂等。
- checkpoint 后崩溃：下次进入下一批。

### 8.5 导入期间推荐状态

导入期间推荐仍然可以展示，但需要明确状态：

- 当前三卡可以先使用导入前的稳定状态。
- 如果用户操作当前三卡，仍走后台统一动作。
- 对导入中新书签的推荐，可以等该批完成后进入后续推荐。
- 搜索可以看到已进入永久池的部分。
- 未完成批不承诺可搜索、可推荐。

### 8.6 导入完成后的恢复动作

所有批完成后：

1. 标记任务完成。
2. 确认记录池已经补齐。
3. 确认 S 基础数据可用于推荐。
4. 触发必要的 S 分批计算。
5. 刷新或延迟刷新当前三卡。
6. 清理完成任务的临时状态。

### 8.7 失败处理

如果某一批连续失败：

- 不静默失败。
- 记录失败原因。
- 标记任务需要继续或需要用户重试。
- 推荐页面可以显示“书签更新未完成”的状态。
- 已完成批保持有效。

### 8.8 验收标准

- 导入大量书签时，任务能按批推进。
- 崩溃后重新打开，能从 checkpoint 继续。
- 当前批重跑不会重复污染记录池或 S 池。
- 已完成批不会丢。
- 导入未完成时，已有推荐状态仍可用。
- 导入完成后，新书签能进入后续推荐候选。

## 9. 删除项：独立 S 缓存健康检查

本轮明确不做独立 S 缓存健康检查。

原因：

- 它容易变成一个模糊的全局扫描。
- 搜索不应该承担修全局池的职责。
- 真正需要的是动作后更新单个书签，批量导入后按 checkpoint 分批补齐。

保留的能力：

- 单个书签动作后更新对应 S。
- 搜索展示诊断信息。
- 批量导入 / 大批量变更按任务补齐。
- 旧数据兼容可以放进批量任务或后台动作里处理。

不再做：

- 不新增“S 缓存健康检查”主入口。
- 不让搜索直接修全局 S 池。
- 不把旧缓存问题单独拉成主工程。

## 10. 执行阶段

### 阶段 1：状态写入盘点

目标：

- 找出所有会影响推荐结果的前台直写。
- 区分业务状态和纯 UI 状态。

产出：

- 推荐业务状态清单。
- 前台直写迁移清单。
- 保留前台状态清单。

完成标准：

- 明确哪些状态必须后台最终写。
- 明确哪些状态可以继续留在前台。

### 阶段 2：后台动作模型

目标：

- 定义统一后台动作。
- 主页面、侧边栏、widgets、快捷复习、搜索按钮都使用这套动作。

产出：

- 动作列表。
- 每个动作的输入、输出、失败语义。
- 哪些动作会影响当前三卡。
- 哪些动作只影响单个书签。

完成标准：

- 同一个动作在不同入口下结果一致。

### 阶段 3：三卡原子化和版本号

目标：

- 选卡、补卡、游标推进、当前三卡写入、版本更新在同一个后台动作里完成。

产出：

- 当前三卡版本字段。
- 过期动作处理规则。
- 补卡规则。

完成标准：

- 两个页面同时刷新，不互相覆盖。
- 快捷复习和页面点击同时发生，不重复处理。

### 阶段 4：搜索动作接入

目标：

- 搜索卡按钮复用推荐卡片后台动作。
- 搜索不自动替换当前三卡。

产出：

- 搜索卡按钮规则。
- 当前卡 / 非当前卡差异处理。
- 打开、打开并复习、标记已复习的语义区分。

完成标准：

- 搜索里操作当前卡，会同步当前三卡。
- 搜索里操作非当前卡，不主动打乱当前三卡。

### 阶段 5：大批量 checkpoint 任务

目标：

- 导入 / 大批量变更按批处理并可恢复。

产出：

- 批量任务状态。
- checkpoint 写入规则。
- 启动恢复规则。
- 失败重试规则。

完成标准：

- 当前批可丢。
- 已完成批不可丢。
- 下次打开能继续。

### 阶段 6：移除前台 fallback 直写

目标：

- 前台动作失败时不再自己补写 storage。

产出：

- 失败提示。
- 重试入口。
- 刷新当前状态入口。

完成标准：

- 推荐业务状态只有后台最终写。

### 阶段 7：回归验证

目标：

- 验证整条链路稳定。

测试场景：

- 主页面和侧边栏同时刷新三卡。
- 多窗口同时点击同一张卡。
- 快捷复习和页面按钮同时触发。
- 搜索里提前复习当前卡。
- 搜索里提前复习非当前卡。
- 搜索里屏蔽当前卡。
- 搜索里屏蔽非当前卡。
- 大量导入书签后立即打开推荐页。
- 导入中崩溃后重启继续。
- 当前批永久池写入后、checkpoint 前崩溃。
- checkpoint 后崩溃。

完成标准：

- 三卡不乱跳。
- 状态不互相覆盖。
- 导入恢复可继续。
- 搜索不打乱当前轮次。

## 11. 风险和取舍

### 11.1 后台动作失败会更明显

以前前台 fallback 可能把失败掩盖掉。

改完后，后台失败会直接展示失败或重试。

这是必要取舍，因为隐藏失败比暴露失败更容易造成状态污染。

### 11.2 checkpoint 会增加工程量

大批量任务要多维护任务状态、批次、失败原因、恢复入口。

这是为了换取导入恢复的确定性。

### 11.3 版本检查可能让旧页面操作失败

旧页面拿着过期三卡版本操作时，后台会拒绝。

前台需要刷新状态。

这是为了避免旧请求覆盖新状态。

## 12. 最终执行顺序

推荐顺序：

1. 盘点推荐状态前台直写。
2. 定义后台统一动作。
3. 实现三卡原子化和版本号。
4. 接入快捷复习。
5. 接入主页面、侧边栏、widgets。
6. 接入搜索卡按钮。
7. 去掉前台 fallback 直写。
8. 实现大批量 checkpoint 任务。
9. 做多入口和导入恢复回归。

## 13. 最终验收口径

本轮完成后，应该能回答这几个问题：

1. 同一个书签在主页面、侧边栏、搜索、快捷键里操作，最终规则是否一致？
2. 两个页面同时刷新三卡，旧结果是否还能覆盖新结果？
3. 搜索里提前复习一个书签，会不会打乱当前三卡？
4. 搜索里屏蔽当前卡，当前三卡是否能正确补卡？
5. 大批量导入中浏览器崩溃，已完成批是否还在？
6. 下次打开后，未完成批是否能继续？
7. 前台是否还会偷偷写推荐业务状态？

只有这些都通过，才算真正解决本轮最初问题。
