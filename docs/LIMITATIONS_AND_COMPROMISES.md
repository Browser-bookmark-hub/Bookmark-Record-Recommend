# Bookmark Record and Recommend 妥协与限制说明

整理基准：2026-07-08

本文档只记录当前代码和浏览器平台带来的限制。旧计划里的“当前”“最终”“必须”等表述不应绕过代码复核直接引用。

## 1. 浏览历史不是无限完整事实源

- Chrome History API 受浏览器自身保留策略、用户清理历史、搜索分页上限和事件粒度影响。
- 当前代码维护本地浏览历史缓存，并通过后台同步、失败重试和局部修复降低缺口。
- 但浏览器未保留、用户已删除且 API 不再返回的数据无法凭空恢复；当导出或同步无法证明完整覆盖时，应按“部分数据”理解。

## 2. 活跃时间是近似统计

- 活跃时间捕捉依赖标签页、窗口焦点、idle、可见性、URL 变化和 Service Worker 恢复等事件。
- 多窗口、系统休眠、标签页冻结、浏览器回收 Service Worker 时，状态可能短暂滞后。
- 排行适合作为使用强度信号，不适合作为精确工时或审计数据。

## 3. 推荐分数是本地启发式结果

- 当前推荐主链路基于本地 S 值缓存和行为信号，不是服务端模型。
- `SCORE_ALGO_VERSION` 当前为 `6.4`，历史文档中 v6.2/v6.3 的规则只能作为演进来源。
- 用户的跳过、屏蔽、待复习、翻卡等操作会影响推荐，不应把分数理解为稳定、客观、全局排序。

## 4. 语言切换与新增语言需要重构或共建

- 项目最开始只规划了中文和英文两套界面文案，当前语言切换也是围绕 `zh_CN` / `en` 两个值实现。
- Manifest 描述使用了浏览器 `_locales`，但主界面并不是完整、统一的 i18n 系统。`history_html/history_i18n.js` 里有集中维护的 `i18n` 对象，同时 `history_html/history.js`、`history_html/search/search.js`、`review_reminder/review_reminder_popup.js` 等文件仍有大量 `currentLang === 'zh_CN' ? ... : ...`、`currentLang === 'en' ? ... : ...` 和独立文案表。
- 因此新增繁体中文、法语、俄语、西班牙语、阿拉伯语、日语、韩语等语言，不是只添加 `_locales/<lang>/messages.json` 就能完成。可先用 Python 或脚本整理、提取硬编码文案，但真正麻烦的是替换索引、补齐翻译入口、验证每个动态渲染路径。
- 不同语言文字长度、换行方式和阅读方向差异明显。法语/俄语/西班牙语可能撑开按钮和标签，日语/韩语需要检查紧凑布局，阿拉伯语还涉及 RTL 方向、图标位置、表格和弹窗对齐等专项调试。
- 后续可走两条路线：要么重构语言系统，把主界面、弹窗、搜索、推荐、同步视图等统一到可扩展 i18n；要么维持现状，逐个搜索 `zh_CN` / `en` 和硬编码中文英文来补语言。前者工程量更大但长期更稳，后者适合社区共建小步补齐，但每种语言都需要 UI 回归。

## 5. 当前没有覆盖性自动化测试套件

- CodeGraph 对关键符号提示“未发现覆盖测试”。
- `docs/归档/18--冒烟测试清单-历史手测.md` 可作为手动回归参考，但不是自动化测试规范。

## 6. 当前存在明显架构债

- 部分 JavaScript 文件过大，整体架构和功能边界早期规划不足。
- 大文件之间耦合较深，功能、状态、渲染、存储、同步、事件处理互相嵌套，容易出现“牵一发而动全身”的修改风险。
- 后续开发应适当引入模块化，把全局变量、存储层、状态流转、后台任务和 UI 渲染边界顶住，再继续扩展功能。
- 当前 AI 提示、注释和局部约束的方式已经能降低维护成本，但不能替代稳定的模块边界与架构治理。
- 相关历史参考：commit `48f3475621896c3b7f0fcf152a0304c1180a46d8`。

## 7. 浏览器 API 与平台能力存在硬限制

### 7.1 书签点击可能产生重复浏览记录

- 观察到点击一个书签时，浏览器历史记录有时可能出现两条访问记录。
- 这类问题来自浏览器历史记录层或官方 API 暴露的数据行为，插件侧只能做去重、合并和容错，不能保证源头只产生一条记录。
- 如果后续要严谨描述该问题，应再核验 Chrome / Chromium issue、扩展开发者反馈或官方文档中的相关记录。

### 7.2 无法获取用户命名的窗口名称

- 浏览器扩展 API 能拿到窗口 ID、类型、焦点、状态等信息，但没有可用 API 获取用户给窗口设置的名称。
- 因此插件不能可靠记录或展示“窗口名称”，只能使用窗口 ID、标签页、URL、标题、时间等可获得信息做关联。

### 7.3 历史删除事件只有 URL，缺少精确时间

- 浏览器历史读取的精确度较高，例如拉取历史访问记录时能拿到较清晰的访问数据。
- 但删除事件只返回被删除的 URL 列表或全部清空标记，不返回具体删除的是哪个时间点的访问记录。
- 当前可行处理方式：
  - 浏览器删历史时只给 `urls` 列表，不告诉时间范围。
  - 对这些 URL，先删除本地缓存里的对应记录。
  - 再用 `getVisits(url)` 重新拉一遍，把删除后仍存在的访问记录写回缓存。
  - 如果只是删除一条历史，通常只影响一个 URL，这个操作很轻量。
- 只有在 `allHistory=true`，或删除 URL 数量非常大时，才应考虑清缓存后全量重建。

## 8. 本地 Profile 边界限制未来同步形态

- Bookmark ID 是本地浏览器 profile 专用标识。书签删除后恢复、换浏览器、换设备，ID 都可能变化。
- 因此任何“按 Bookmark ID 保存的状态”都无法保证无损迁移。
- 插件的推荐与记录状态依赖本机 profile 内的行为上下文；换设备后，即使同步部分数据，也不等于同步了另一台机器的真实使用环境。
- 所以该插件未来很可能长期保持本地优先，甚至主要局限于本地 profile 使用；跨设备同步只能作为辅助能力，不能承诺完整一致。
