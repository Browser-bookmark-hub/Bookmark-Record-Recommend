# Bookmark Record and Recommend 妥协与限制说明

整理基准：2026-07-08

本文档只记录当前代码和浏览器平台带来的限制。旧计划里的“当前”“最终”“必须”等表述不应绕过代码复核直接引用。

## 1. 浏览历史不是无限完整事实源

- Chrome History API 受浏览器自身保留策略、用户清理历史和搜索分页上限影响。
- 当前代码已维护本地浏览历史缓存，并有每日后台同步与失败重试，但被用户删除或浏览器未保留的数据无法凭空恢复。
- 当导出或同步无法证明完整覆盖时，应按“部分数据”理解，不应承诺全量长期浏览历史。

## 2. 活跃时间是近似统计

- 活跃时间捕捉依赖标签页、窗口焦点、idle、可见性、URL 变化和 Service Worker 恢复等事件。
- 多窗口、系统休眠、标签页冻结、浏览器回收 Service Worker 时，状态可能短暂滞后。
- 排行适合作为使用强度信号，不适合作为精确工时或审计数据。

## 3. 推荐分数是本地启发式结果

- 当前推荐主链路基于本地 S 值缓存和行为信号，不是服务端模型。
- `SCORE_ALGO_VERSION` 当前为 `6.4`，历史文档中 v6.2/v6.3 的规则只能作为演进来源。
- 用户的跳过、屏蔽、待复习、翻卡等操作会影响推荐，不应把分数理解为稳定、客观、全局排序。

## 4. GitHub 同步依赖用户配置与网络

- GitHub 同步需要用户提供 Token、Owner、Repo、Branch 等配置。
- 推送/拉取受 GitHub API、网络状态、分支权限、Token 权限和速率限制影响。
- 数据会写入用户配置的仓库路径；仓库可见性和协作者权限由用户自行管理。
- Token 保存在浏览器本地存储中，应按敏感配置处理。

## 5. Markdown 渲染与编辑是扩展内能力

- 当前页面使用本地 vendor 库渲染和清洗 Markdown，适合展示 AI 结果和任务文档。
- 这不是完整 IDE，也不保证兼容所有 Markdown 扩展语法。
- 文档同步冲突仍应以 GitHub 远端状态、同步配置和本地编辑时间共同判断。

## 6. Favicon 可能不稳定

- Manifest 声明 `favicon` 权限，页面也存在本地缓存和回填逻辑。
- favicon 受站点可用性、浏览器缓存、跨域、公共图标源和本地失败缓存影响。
- 图标只应作为辅助展示，不应作为书签身份的唯一依据。

## 7. Manifest V3 后台不是常驻进程

- `background.js` 是 MV3 service worker，浏览器可按需唤醒和回收。
- 代码通过 alarms、storage 和恢复逻辑降低影响，但不能按传统常驻后台页理解。
- 长任务、批量同步和推荐刷新都需要考虑中断、限频和恢复。

## 8. 当前没有覆盖性自动化测试套件

- CodeGraph 对关键符号提示“未发现覆盖测试”。
- `docs/归档/18--冒烟测试清单-历史手测.md` 可作为手动回归参考，但不是自动化测试规范。
