# AI 同步与 Markdown 体验：最小改动最大收益计划（2026-04-21）

状态：执行计划（v1）  
日期：2026-04-21  
范围：仅 `Bookmark-Record-Recommend`（不改外部参考项目）

## 1. 目标与原则

### 1.1 目标
在不推翻现有架构的前提下，提升同步区与 Markdown 编辑体验，优先保证稳定、可维护、低回归风险。

### 1.2 原则
1. 先小改，后增强，不做大重构。  
2. 数据结构与推拉协议优先保持兼容。  
3. UI 改动和内核改动解耦，分阶段上线。  
4. 每阶段必须可独立验证、可回滚。

---

## 2. 现状判断（冻结）

1. 同步实验区已具备：拉取/推送/同步、文件列表、查看与编辑。  
2. 当前编辑内核是 `textarea`，阅读渲染是 `marked + sanitize`。  
3. 书签画布侧已有自研编辑能力，整体可用，但维护成本高于通用内核。  
4. 当前最可控的收益点是“同步区体验提升”，不是“全项目整体替换”。

---

## 3. 非目标（本计划不做）

1. 不一次性重做书签画布全部编辑系统。  
2. 不引入多套重型编辑器并行运行。  
3. 不改动现有同步数据包协议（JSON/JSONL 主链）。  
4. 不做自动轮询同步和多 Provider 扩展。

---

## 4. 最小改动方案（按优先级）

## 4.1 P0：同步区“体验增强，不换内核”
目标：在保持 `textarea` 的情况下，先补齐高频编辑工具与一致渲染。

动作：
1. 新增轻量工具栏（加粗、斜体、标题、列表、任务列表、引用、代码块、链接、wikilink、callout 模板）。  
2. 工具栏只操作当前编辑文本，不改存储模型。  
3. 阅读渲染规则与现有 `obsidian-markdown` 对齐（确保查看一致）。  
4. 保持现有保存、推送、拉取、重命名、删除流程不变。

收益：
1. 用户体感立刻提升。  
2. 几乎不触碰同步核心逻辑，回归风险最小。

## 4.2 P1：同步区“单实例编辑内核升级”（可选）
目标：仅替换同步区输入层为单实例 CM6，继续复用现有数据流。

动作：
1. `textarea` -> 单实例编辑器（同一时刻仅一个实例）。  
2. 保持 `syncDocsState`、Push/Pull 路径、保存逻辑不变。  
3. 工具栏按钮映射为编辑器命令，不改按钮布局语义。  
4. 阅读渲染仍走既有安全渲染链。

收益：
1. 解决选区、撤销、输入法等底层稳定性问题。  
2. 为后续增强打基础，但改动面仍可控。

## 4.3 P2：画布“编辑态挂载策略”（后续）
目标：不动画布常态展示，仅在进入编辑时挂载编辑内核。

动作：
1. 普通态继续用当前轻渲染。  
2. 仅编辑态挂载，退出即卸载或冻结。  
3. 分批迁移（先空白 MD 卡片，再扩展到其它可编辑块）。

收益：
1. 控制性能成本。  
2. 避免一刀切替换导致交互回归。

---

## 5. 验收标准（每阶段必须满足）

## 5.1 P0 验收
1. 工具栏操作后文本结果正确，保存后重开一致。  
2. 推送与拉取结果不受影响。  
3. 查看态渲染与编辑预期一致，无明显 XSS 风险。  
4. 双语文案（中/英）不破坏现有逻辑。

## 5.2 P1 验收
1. 编辑器替换后，保存、推送、拉取、重命名、删除均通过。  
2. 常见中文输入与撤销重做稳定。  
3. 横向窄宽与双列布局不出现新增滚动回归。  
4. 无新增全局样式污染。

## 5.3 P2 验收
1. 画布帧率与交互无明显下降。  
2. 仅编辑态存在内核实例，不出现多实例堆积。  
3. 迁移卡片类型可回滚到旧实现。

---

## 6. 风险与回滚

1. 风险：编辑器接入引发布局或快捷键冲突。  
回滚：保留旧输入层开关，遇问题可切回 `textarea`。

2. 风险：渲染规则调整导致历史文档显示差异。  
回滚：保留旧渲染函数，支持灰度切换。

3. 风险：工具栏插入规则污染既有文档格式。  
回滚：工具栏功能逐项开关，先灰度放量。

---

## 7. 执行顺序（建议）

1. 本周：完成 P0（最小改动，最大体感收益）。  
2. 下周：评估并选择是否进入 P1（单实例内核升级）。  
3. 稳定后：再进入 P2（画布编辑态挂载）。

---

## 8. 一句话决策

先做 P0，不大改；  
P1 只改同步区输入层；  
P2 最后再碰画布内核。
