# Favicon 子域名通解计划书（Record & Recommend）

日期：2026-04-14  
范围：`Bookmark-Record-Recommend` 全部 Favicon 获取/缓存入口  
目标：统一为“子域名键 + 永久存储优先 + 批量查询 + 单次网络去重”

## 1. 设计原则（通解）
1. 缓存主键统一为 `hostname`（子域名级），不使用 `title`，不使用完整 URL。
2. 永久存储（IndexedDB）为主缓存；渲染和业务判断以 IDB 结果为准。
3. 网络仅用于 IDB 未命中时；成功后写回 IDB。
4. 失败记录使用短 TTL（建议 5 分钟）防止请求风暴。
5. 全项目仅保留一个并发去重表（`inflight`），同 `hostname` 同时最多一个网络请求。
6. 速度优先：前台渲染路径不阻塞等待 `await`，允许偶发 miss 后进入瀑布流并回写覆盖。

## 2. 统一流程
1. 输入 URL。
2. 解析 `hostname`。
3. 先查 `failures(hostname)`，若在 TTL 内则直接返回占位图。
4. 同步返回当前可用图标（命中则用图标，未命中先占位），不等待异步任务完成。
5. 后台异步查 `favicons(hostname)`（IDB）并尝试回填。
6. 若仍缺失或并行策略要求，进入瀑布流抓取。
7. 瀑布流成功：直接写入 `favicons(hostname)`（允许覆盖回写），并清理失败记录。
8. 瀑布流失败：写入 `failures(hostname, ts)`。

## 2.1 快速优先执行细则
1. 默认 `last-write-wins`：后到结果覆盖前结果，不做阻塞式质量比较。
2. 允许偶发“缓存已存在但仍触发网络”并回写覆盖，优先保证响应速度。
3. 保留 `inflight(hostname)` 去重，避免同域瞬时风暴。

## 3. 批量查询要求（性能关键）
1. 对同一批待渲染 URL，先去重为 `hostname` 集合。
2. 以“单个 readonly 事务”批量 `get`，禁止“一条 URL 一个事务”。
3. 未命中的 `hostname` 再批量进入网络层，并使用 `inflight` 去重。
4. 批量写入用单个 readwrite 事务。

## 4. 本项目落地点
1. `history_html/history.js` 的 FaviconCache 查询链路统一为上述流程。
2. `getFaviconUrl` 等同步渲染入口仅负责占位图与回填，不再绕过 IDB 主链路。
3. 日历/推荐卡片相关入口统一调用批量解析方法（内部仍是通用算法，不做页面特化）。

## 5. 限制与风险
1. 首次冷启动仍可能慢：因为 IDB 为空，必须走网络。
2. 源站无图标或返回默认占位图时，命中质量受源站限制。
3. 子域名级键会增加条目数（相对主域名级），但可避免“不同子域误复用”。
4. IndexedDB 异步 I/O 天然有开销；必须通过批量事务降低成本。
5. 采用覆盖回写策略时，极少数情况下可能出现“旧图被新图替换”的抖动。

## 6. 性能结论（可预期）
1. 二次打开/二次渲染会显著加快（命中 IDB）。
2. 性能瓶颈不在“是否查 IDB”，而在“事务数量是否过多”。
3. 采用批量事务后，规模增长主要影响网络 miss，不会线性拖慢缓存命中路径。

## 7. 验收标准
1. 同一 `hostname` 多次渲染，网络请求次数趋近于 0（除 TTL 到期与手动刷新）。
2. 控制台统计可区分：`idb_hit / network_fetch / failure_ttl_block`。
3. 在常见页面刷新中，绝大多数图标由 IDB 命中返回。

## 8. 回滚策略
1. 保留旧链路开关（功能标志），出现兼容问题可切回旧流程。
2. 若发现 miss 异常增多，先回滚失败 TTL 与批量写入改动，再定位源头。
